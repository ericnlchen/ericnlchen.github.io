L.ink

interactive L-system drawing tool

Brown University HCI Lab | Advisor: Jeff Huang

02.2024—now

Exploring how controllability and surprise impact the creative experience of procedural illustrators

Idea
I joined the Brown HCI Lab with an idea inspired by my interest in the generative grammars known as L-systems...

In an L-system, the user has no control beyond specifying the initial state. As a result, they're detached from the growth process

I wanted to make the entire growth process an interactive experience by making L-systems into a procedural illustration tool

What new stories can be told with living, growing ink?

Implementation
interactive growing ink
Implementation

L-systems are notoriously difficult to author due to their unpredictable nature. There's a theoretical reason for this--it's an NP-complete problem.



I got around this theoretical boundary with inspiration from Bret Victor's guiding principle: creators need an immediate connection to what they create.



With live visual feedback, users can make incremental adjustments toward their desired ink style.

visual ink-editor with live preview
Outcomes/Future

L.ink places the artist and their procedural ink in a continuous feedback loop.

Unpredictable ink growth affects the artist's hand movement

Artist's hand movement determines ink growth

Through L.ink, I'm studying how digital illustrators' experience is impacted by the dynamics of this feedback loop.


I designed a formal user-study to run with Rhode Island School of Design students next February.


I'm preparing an extended abstract in progress for CHI 2025 Late-Breaking Work and a first-author paper for UIST 2025

Pull

live autoencoding with interactive constraints

Rhode Island School of Design: Drawing with Computers + Personal Project

11.2024—12.2024

Exploring the tensions in representations of meaning

Idea

This project started with the observation that adjacency matrices provide a natural connection from images to graphs. Can the data of an image be encoded within the tensions of a network?

If so, could we create an interactive system where pulling on the network of nodes deforms and contorts the image?

I imagined this as an almost grotesque experience of forcefully pulling apart the meaning of something.

Implementation
pulling on latent graph

After some experimentation I found that encoding raw pixel values didn't create the effect I had in mind—distortions were local and rectilinear.


I wondered if embedding in the latent space of a neural network would create a more interesting "semantic" distortion, so I trained an autoencoder on the (tiny) ImageNet dataset and wrote a Python backend to live decode the link lengths of a D3.js graph simulation.


I also decided to try using the webcam feed as input rather than a still image, to create a mirror-like experience. Here are the results using two different autoencoders that I trained:

SSIM Loss

MSE Loss

Implementation
faces

I still wanted to create a version where pulling on nodes distorted the image in a more meaningful way. Learning a latent space over a hugely diverse range of natural images was proving tough, so I decide to focus on a dataset containing only faces (CelebA).


I experimented with training my prior autoencoder but ended up finding more success with a pre-trained variational autoencoder, which was able to learn a smoother latent space.


For this version, I wanted to visualize the latent variables as a set of audio bars instead of encoding them in graph. Tuning the green bars up or down scales latent variables, allowing the user to morph their face freely.


Visual Code Studio

knockoff gallery of bona-fide art

Rhode Island School of Design: Drawing with Computers + Personal Project

10.2024—12.2024

Re-interpreting source code as drawing instructions

Idea

A variety of existing systems empower us to transform code into drawings. P5.js, Processing, and openFrameworks are intended to enable visual creation. But visual complexity is hidden within code, regardless of intention. What if we could build a system to map arbitrary source code to visual representations?


My goal was to reinterpret utility as wonder.

Implementation

I conducted a variety of experiments around the theme of misinterpreting source code as drawing instructions.


My first idea was to create a kind of concrete poetry that procedurally generates itself—in other words, the source code text is used both as procedural instructions for where to place content AND it also is the content.


I decided to pivot because I wanted to establish a direct connection between a single instruction and source code and a single drawing instruction, like: draw a stroke from A to B.


To this end, I wrote a program to extract Python bytecode from any .py file, allowing me to map each bytecode instruction to a visual operation. I settled on using pixel-wise operations—LOAD_CONST moves vertically and places a pixel, CALL_FUNCTION changes the current drawing color, etc. The parameters of the bytecode instructions become parameters governing how much to change color or how far to move.

CV for Graphics and Interaction | Team: Eric Chen, Josh Yang, Serena Pulopot, Zihan Zhu

11.2024—12.2024

Enhancing the materiality of VR drawing applications

Idea

In class, we surveyed recent state-of-the-art papers on 3D Gaussian Splatting. Inspired by the analogy between 2D pixels and 3D Gaussians, I proposed a project to create a 3D Gaussian VR painting tool. I saw massive room for improvement over existing VR painting tools, where strokes are awkward-looking "ribbons" and "tubes."


My proposal was accepted by the class and my team and I got to work.
